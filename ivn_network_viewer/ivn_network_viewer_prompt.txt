INTEGRATED VALUE NETWORK (IVN) VISUALIZER - LEARNING MODEL TRAINING PROMPT

================================================================================
PROBLEM STATEMENT
================================================================================

Organizations need to understand complex dependency relationships between components in their Integrated Value Network. When components enable or depend on other components across multiple sources, stakeholders struggle to visualize these relationships, identify patterns, filter relevant connections, and understand the flow of dependencies at both granular (component) and high-level (source) perspectives.

================================================================================
BROADER CONTEXT
================================================================================

In enterprise architecture and organizational planning, an Integrated Value Network represents how different components (capabilities, services, processes, systems) interact and depend on each other. Understanding these relationships is critical for:

- Impact analysis: What happens if component X fails or changes?
- Strategic planning: Which components are foundational vs. derivative?
- Resource allocation: Where should investments focus?
- Risk assessment: What are single points of failure?
- Communication: How to explain complex interdependencies to stakeholders?

The data comes from normalized Excel sheets tracking components, their relationships (alignments), and their originating sources. This script transforms tabular dependency data into interactive visual networks.

================================================================================
CONCEPTS THAT MUST BE UNDERSTOOD
================================================================================

1. DIRECTED GRAPHS (DiGraph)
A mathematical structure where nodes are connected by edges with direction. In our context:
- Nodes = Components or Sources
- Edges = Dependency relationships (A enables B means edge from A to B)
- Direction matters: enabling → dependent

2. GRAPH VISUALIZATION
Converting abstract graph structures into visual layouts where:
- Position is calculated algorithmically (spring layout)
- Nodes are rendered as markers
- Edges are rendered as lines
- Interactive elements provide additional information on hover

3. DATA NORMALIZATION
Separating data into related tables to eliminate redundancy:
- Components table: Unique components with IDs and descriptions
- Alignments table: Relationships between components
- Sources table: Origin sources for components

4. DATA MERGING
Joining multiple tables to create enriched dataset with all needed information in one place.

5. STREAMLIT WEB APPLICATIONS
Building interactive web-based data applications with Python where:
- UI elements are created with simple function calls
- State management is handled automatically
- Caching optimizes performance
- Updates trigger automatic reruns

6. FILTERING AND INTERACTIVITY
Allowing users to narrow down displayed data based on text matching against descriptions.

DOMAIN-SPECIFIC TERMS
================================================================================

- IVN (Integrated Value Network): The complete system of interconnected components
- Component: An individual capability, service, or element within the network
- Enabling Component: A component that provides capabilities to another component
- Dependent Component: A component that relies on another component
- Alignment: A documented relationship showing one component enables another
- Source: The originating document, standard, or authority defining components

- Edge: Visual representation of a dependency relationship

================================================================================
HIGH-LEVEL DESCRIPTION OF WHAT THE SCRIPT DOES
================================================================================

This script creates an interactive web-based visualization tool that:

1. Loads and merges data from three Excel sheets (Components, Alignments, Sources)
2. Creates enriched dataset with both enabling and dependent component information
3. Provides text-based filters to narrow down displayed relationships
4. Generates two interactive network diagrams:
   - Component-Level Network: Shows individual component dependencies
   - Source-Level Network: Shows high-level source dependencies
5. Renders networks using force-directed layout for aesthetic node positioning
6. Enables hover interactions to see component descriptions
7. Uses tabs for easy switching between network views

================================================================================
KEY CONCEPTS AND LEARNING DATA
================================================================================

CONCEPT 1: Graph Construction from Relational Data

The transformation process:
1. Alignments table has: enabling_component_id, dependent_component_id
2. Each row becomes a directed edge in the graph
3. NetworkX DiGraph stores these relationships
4. Graph structure enables layout algorithms

Learning pattern:
```
FOR each alignment row:
    CREATE edge from enabling_component_id TO dependent_component_id
    ADD edge to directed graph
```

CONCEPT 2: Data Enrichment Through Multiple Merges

Starting with alignments, we sequentially merge:
1. Left merge with components on enabling_component_id → get enabling description & source_id
2. Left merge with components on dependent_component_id → get dependent description & source_id
3. Left merge with sources on enabling_source_id → get enabling source name
4. Left merge with sources on dependent_source_id → get dependent source name

Result: Single DataFrame with all information needed for filtering and display

CONCEPT 3: Force-Directed Layout

Spring layout treats:
- Edges as springs pulling connected nodes together
- Nodes as charged particles repelling each other
- Iterative simulation finds equilibrium positions
- Seed parameter ensures reproducible layouts

CONCEPT 4: Plotly Graph Visualization

Two-trace approach:
1. Edge Trace: Line segments connecting node positions
   - Constructed by alternating between start/end coordinates with None separators
   - None creates line breaks between edges
2. Node Trace: Scatter points at node positions
   - Markers for visual representation
   - Text labels for identification
   - Hover text for detailed information

CONCEPT 5: Streamlit Interactivity Pattern

Flow:
1. User inputs trigger script rerun from top
2. Cached data (@st.cache_data) avoids reloading Excel
3. Filters apply to DataFrame
4. Graphs regenerate with filtered data
5. UI updates automatically

- Missing values in enabling/dependent component IDs will cause those alignments to be filtered out

File Location:

- Must be named exactly "IVN_Normalized.xlsx"

USER INPUTS (during runtime):

2. Dependent Filter (Sidebar Text Input)
   - Purpose: Filter alignments by dependent component description
   - Type: Text string (case-insensitive)

   - Empty = show all

Filters are AND combined: both must match if both are provided.

ERROR HANDLING LOGIC
================================================================================

CURRENT ERROR HANDLING:

The script has MINIMAL explicit error handling. Failures propagate as exceptions:

1. FILE NOT FOUND
   - Trigger: IVN_Normalized.xlsx not in script directory
   - Behavior: Streamlit displays error page with stack trace
   - User action: Place file in correct location, refresh browser

2. SHEET NOT FOUND
   - Trigger: Excel file missing Components, Alignments, or Sources sheet
   - Behavior: Pandas raises ValueError
   - User action: Verify sheet names match exactly

3. MISSING COLUMNS
   - Trigger: Expected columns not in sheets
   - Behavior: KeyError during merge or access operations
   - User action: Verify Excel schema matches requirements

4. EMPTY DATA
   - Trigger: No rows in alignments or all filtered out
   - Behavior: Empty graphs display (valid outcome)
   - User action: Adjust filters or add data

5. INVALID REFERENCES
   - Trigger: component_id or source_id references that don't exist
   - Behavior: NaN values in merged DataFrame (handled by dropna)
   - User action: Fix data integrity issues in Excel

FAILURE MANAGEMENT:

The script does NOT gracefully handle errors. Production version should add:
- Try-except blocks around file operations
- Validation of Excel schema before processing
- User-friendly error messages
- Fallback behaviors
- Logging
