ivn_populate_component_descriptions_from_URLs.py: Context and LLM Training Specification

---

Problem Explanation:
This script automates the enrichment of an Excel workbook (ivntest.xlsx) containing two parallel sets of components: Enabling and Dependent. Each component has a name, description, and URL. The script's goal is to populate missing or placeholder descriptions by fetching, cleaning, and validating content from the provided URLs, and to track the status of each fetch. It also consolidates duplicate records based on a set of key columns, keeping the most detailed information in certain fields.

Broader Context:
In large organizations, component registries or inventories often require up-to-date, meaningful descriptions for each item. Manual population is error-prone and time-consuming. Automating this process ensures consistency, auditability, and efficiency, especially when descriptions can be programmatically derived from authoritative online sources.

Concepts to Understand:
- Data enrichment and validation
- Web scraping and content extraction (HTML, PDF, text, JSON, XML)
- Data cleaning and normalization
- Error handling and retry logic
- Caching to avoid redundant network requests
- DataFrame manipulation and grouping in pandas
- Excel file I/O
- Progress reporting and runtime logging
- Record consolidation based on key fields

Prerequisite Knowledge:
- Python programming
- pandas for data manipulation
- requests for HTTP operations
- BeautifulSoup for HTML parsing
- PyPDF2 for PDF text extraction
- Basic understanding of Excel file formats
- Exception handling in Python

Domain-Specific Terms:
- Enabling Component: A system or process that enables another (dependent) component.
- Dependent Component: A system or process that depends on an enabling component.
- Data enrichment: The process of enhancing data with additional information.
- Fetch Status: The result of attempting to retrieve and process content from a URL.

High-Level Description:
1. Reads ivntest.xlsx from the script directory.
2. For each row, checks if the Enabling/Dependent Component Description is missing or a placeholder. If so, fetches content from the corresponding URL, cleans and validates it, and updates the description.
3. Adds status columns to track the outcome of each fetch.
4. Caches URL fetches to avoid redundant requests.
5. Cleans and normalizes extracted content, enforcing minimum content length and rejecting trivial or invalid results.
6. Handles errors and retries with exponential backoff for network issues.
7. Truncates descriptions to Excel-safe length.
8. Consolidates duplicate records based on a set of key columns, keeping the row with the most detailed information in the description and office fields.
9. Saves the enriched data to a timestamped Excel file and logs runtime.
10. Outputs a failure report for rows with errors or insufficient content.
11. Prints progress, ETA, and summary statistics.

Why the Script Exists:
To automate and standardize the process of populating and validating component descriptions in a registry, ensuring data quality and reducing manual effort.

Input/Output Definitions:
- Input: ivntest.xlsx (Excel file with component data; must contain required columns for both Enabling and Dependent pairs).
- Output: component_descriptions_from_URLsYYYYMMDDHHMM.xlsx (enriched Excel file), ivn_populate_descriptions_runtime.txt (runtime in seconds), failed_url_reportYYYYMMDDHHMM.csv (failure report).
- Output columns added: Enabling Fetch Status, Dependent Fetch Status.
- Output is validated for content length, non-triviality, and Excel compatibility.

Structure and Content of Final Output:
- All original columns preserved.
- Description columns updated only if enrichment is required.
- Status columns indicate the result of each fetch/validation.
- Duplicate records (based on key columns) are consolidated, keeping the row with the most detailed information in the description and office fields.
- Output Excel file is truncated to 32,000 characters per cell for safety.

Validity Checks:
- Description must not be empty, must not simply repeat the URL, and must meet minimum length.
- Fetch status must be tracked and reported.
- Output file must be saved successfully; fallback to CSV if Excel fails.
- Duplicate records must be consolidated as specified.

Definitions of a Successful Outcome:
- All required columns are present in the input.
- All rows are processed, with descriptions enriched where needed.
- All fetches are validated and statuses recorded.
- Output file is saved and contains consolidated, enriched data.
- Failure report is generated for any errors or insufficient content.
- Progress and summary statistics are printed.

Error Handling Logic:
- Retries on timeouts, 5xx, and 429 errors with exponential backoff.
- Distinguishes between different error types (403, 404, 429, connection errors, redirect loops).
- Caches results to avoid redundant fetches.
- Logs failures and outputs a CSV report.
- Falls back to CSV output if Excel write fails due to illegal characters.
